What Rutter charts

The complete doc set, in one namespaced home.

A short contract at the root, depth under .agent/, and thin pointers wherever a tool demands its own file. Identical structure in every project — greenfield or ten years old.

The one rule

One canonical home. Pointers everywhere else.

Content lives in exactly one place. Tool-specific locations only ever contain a pointer to it — so drift isn’t discouraged, it’s structurally impossible. One rule, applied everywhere, easy to keep.

  • CLAUDE.md · GEMINI.md──›AGENTS.md
  • .cursor · .github──›AGENTS.md
  • FEATURE.md links──›references/ · docs/
The passage

You stay at the helm.

Rutter never sails ahead of you. It detects your project silently, then works task by task — every document proposed before it’s written, with proceed, adjust, or skip at each step. Autonomous tools deliver surprises; Rutter delivers exactly what you approved.

STEP 00

Detect

Silent and automatic: code, manifests, git history present? Existing mode — it documents what’s there. Empty folder? Greenfield — a short intake, then docs from your idea. Its conclusion is stated in the very first proposal, so a wrong guess dies immediately.

STEP 01

Propose

“Next I’ll generate ARCHITECTURE.md. Based on discovery it will say: <two-line summary>.” Nothing is written yet.

PROCEED · ADJUST · SKIP
STEP 02

Execute

Writes only the approved task’s files, from templates, filled only with confirmed facts. Re-runs touch managed sections — your hand edits survive.

STEP 03

Report

“Done — created X. Next up: SETUP.md.” Then back to Propose for the next document, until the final verification checklist.

LOOPS PER DOCUMENT

Skip is how you shape the output in practice: decline the docs a project doesn’t need, task by task. Existing repos lean on ARCHITECTURE and whichever conditional docs have real evidence; greenfield leans on the FEATURE specs and TASKS the intake seeds — the detected mode sets the defaults, you steer from there.

Draw exactly as much water as your repo needs

No presets to pick — evidence decides, you steer with skip.

Every doc is proposed only when discovery actually found something to base it on. There's no profile to choose up front — but here's the shape that decision typically takes:

A CLI tool or script thin repo

No UI, no test suite, no CI — there's just not much evidence to work with yet.

  • AGENTS.md + adapters
  • — most .agent/docs/ skipped, no evidence
  • — features created only if you ask

A typical app most repos

Real code across the board — tests, UI, maybe a deploy pipeline.

  • AGENTS.md + adapters
  • .agent/docs/ — whichever docs have real evidence
  • .agent/skills/ scaffold

A greenfield idea empty folder

No code yet — a short intake stands in for discovery.

  • Everything a typical app gets
  • .agent/features/ seeded from your idea
  • TASKS.md with stable task IDs
Why “Rutter”

Before charts existed, there were rutters.

A rutter — from the Portuguese roteiro — was a mariner’s hand-written pilot book: hard-won, closely guarded directions that let any captain navigate waters they had never sailed.

Five hundred years later, the problem is the same. Your repo is water an agent has never sailed. Rutter writes down what the pilot knows — so any agent, in any tool, arrives without running aground.
