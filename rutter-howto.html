How to

Your first passage, in five minutes.

This walkthrough runs Rutter on a real repo, end to end — so you see exactly what it asks, what it writes, and where your hands stay on the wheel the whole time. Nothing here is automatic except detection; everything written was approved by you first.

STEP 1

Get Rutter aboard — two ways, same files.

One zip, one START-HERE.md inside. Pick your path:

Install as a skill

Claude Code · install once, use everywhere

Unzip into your skills folder. Rutter now triggers itself whenever you ask for agent docs, in any repo you open.

# global — every project ~/.claude/skills/rutter/ # or per-project <repo>/.claude/skills/rutter/

Drop into a repo

Cursor · Copilot · Gemini CLI · Windsurf · any tool

Unzip straight into the repo you want documented, then paste one sentence into your assistant:

<repo>/rutter/ # then paste: "Read rutter/SKILL.md and follow it against this repository."
Which should you pick? If you live in Claude Code, install the skill — it stays out of your repos entirely. The drop-in folder is for everything else, and it's disposable: delete it after the run or keep it for re-runs.
STEP 2

Watch a real run — an existing repo.

Here's Rutter on an existing web app. Notice the rhythm: propose → your call → execute → report, one document at a time. This transcript is abridged but the shape is exact.

SESSION — my-shop-app/
YOU

Read rutter/SKILL.md and follow it against this repository.

RUTTER

Detection: this looks like an existing project — a Next.js 14 frontend and a FastAPI backend in one repo, pnpm + pip, pytest present, GitHub Actions CI. I'll run in existing mode: document what's here.

Task 1 is discovery: I'll scan the structure and report my findings before writing anything.

PROCEED · ADJUST · SKIP
YOU

Proceed.

RUTTER

Discovery findings — correct anything before I document it:

stack     Next.js 14 (app router) · FastAPI · Postgres
layout    web/ frontend · api/ backend · shared nothing
tests     pytest in api/tests — web/ has none
commands  pnpm dev · uvicorn main:app · pytest
unknown   deployment target — I'll mark it TODO(you)

Task 2: generate AGENTS.md + the four adapters from these facts.

PROCEED · ADJUST · SKIP
YOU

Adjust — deployment is Railway, both services. And add a rule: never touch api/migrations/ by hand.

RUTTER

Noted both — deployment recorded, and the migrations rule goes in AGENTS.md under hard rules. Done — created:

AGENTS.md          CLAUDE.md · GEMINI.md
.cursor/rules/     .github/copilot-instructions.md

Task 3: .agent/docs/ARCHITECTURE.md — it will describe the two-service layout, request flow web→api→Postgres, and the directory map.

PROCEED · ADJUST · SKIP
YOU

Proceed. And when you get to TESTING.md, skip it — test strategy is changing next sprint.

RUTTER

Understood — TESTING.md will be skipped. …several tasks later…

Final verification:

✓ AGENTS.md — all 9 links resolve
✓ .agent/docs/ — 4 of 5 (TESTING skipped, by you)
✓ .agent/features/ + skills/ — scaffolded, INDEX matches
✓ no invented facts — 0 TODO(you) left (deployment resolved via your input)

Passage complete. You can delete rutter/, keep it for re-runs, or gitignore it.

The wrong-guess safety net: because detection is announced in the very first proposal, a mistake ("this looks like a Django app" when it's Flask) gets corrected before a single file exists — that's the point of checking in every task instead of once.
STEP 3

No code yet? Greenfield works the same way.

Point Rutter at an empty folder and detection flips to greenfield: instead of scanning, Task 1 becomes a short intake — what you're building, for whom, roughly how. Then the same loop runs, but the defaults shift toward the planning side: .agent/features/ gets seeded with your initial feature folders, each with a FEATURE.md and a TASKS.md of stable-ID tasks. When you later tell any agent “continue Task 1.2”, it knows exactly what you mean — that's your docs working.

STEP 4

After the run — how the docs get used.

Start every agent session the same way

In any tool, any model, the session-opening line is now identical across all your projects:

"Read AGENTS.md. Then continue Task 2.1 of the checkout feature."

It stays current on its own

Finishing a task already keeps AGENTS.md and .agent/docs/ honest — a resolved TODO(you), a new shared class, a real decision recorded — no separate step. Run Rutter again for the bigger stuff instead: a new conditional doc that now qualifies, a stack migration, adapter changes. Either way, only the content between the markers ever gets touched.

FAQ

Short answers to fair questions.

Which assistants does it work with?

Any agent that can read files and follow instructions — Claude Code (native skill), Cursor, Copilot, Gemini CLI, Codex, Windsurf, Aider. There's no tool-specific vocabulary inside, and no network access needed: the whole skill is self-contained markdown.

Does it need to run scripts or install anything?

No. Discovery is just reading files — the same Read/Grep/list-files calls any coding agent already has — so it works identically wherever the skill itself can run. There's no separate scanner to install and nothing extra to fail.

Will it invent things about my project?

No — that's a hard rule. Every statement in the generated docs traces to what discovery found or what you said at a check-in. Anything unknown is marked TODO(you), never guessed — so it's always clear whose job it is to fill the gap.

I don't want all the documents.

Then don't approve them — skip is a first-class answer at every proposal. There's no profile to opt out of in advance; evidence sets the starting defaults (a doc only gets proposed if discovery actually found something to base it on), and the check-ins are where you make the final call, document by document.

How is this different from Keel?

Keel is the autonomous builder for non-coders: one idea, one confirmation, a finished app. Rutter is the interactive doc generator for developers: task-by-task check-ins, and the output is the documentation set — not a built app. Keel sails for you; Rutter keeps you at the helm.
